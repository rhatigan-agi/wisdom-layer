# v1.2.0 migration — `facts.source_memory_ids` (schema-half)

Status: schema migration only. **No application code change required
for v1.2.0 users.** The migration runs automatically on first boot of
the v1.2.0 wheel against an existing store; it can also be run
explicitly via the new `wisdom-layer-migrate` CLI.

Risk: low. Non-destructive on disk; legacy column retained; idempotent
backfill.

Time: seconds for typical stores; under one minute for 100k facts.

---

## Why

In v1.0 and v1.1 every fact carried a single `source_memory_id`. That
worked while facts were one-to-one with the memory that minted them,
but as memories grow dense and reconsolidation runs, a fact synthesized
from multiple supporting memories collapses to one contributor — every
other supporting reference is silently lost.

In v1.2.0 that loss has a sharper consequence: team insights derive
from facts, and the cross-agent provenance walk
(`agent.provenance.walk_xagent`) needs the full set of contributing
back-pointers. A single-id fact terminates the walk early, exactly
the failure mode the moat is designed to prevent.

The application-level fix lands in **v1.2.1** — that's where
`Fact.source_memory_ids` becomes the primary field and
`agent.facts.trace()` ships. v1.2.0 ships only the schema half so the
on-disk column is ready when v1.2.1 starts populating it; a single
migration step instead of two.

## What ships in v1.2.0

Migration `0024_facts_source_ids` (one file each for SQLite and
Postgres):

- Adds a new column `source_memory_ids` (JSON array on SQLite,
  `text[]` on Postgres) defaulting to `'[]'`.
- Backfills the new column from the legacy scalar:
  - rows where `source_memory_id IS NOT NULL` → `["<that id>"]`
  - rows where `source_memory_id IS NULL` → `[]`
- The backfill is idempotent: it skips rows whose `source_memory_ids`
  is already populated, so a partially-applied migration recovers
  cleanly on rerun.
- Retains the legacy `source_memory_id` scalar column.

In v1.2.0 the SDK **continues to read and write the legacy scalar**
exactly as it did in v1.1.x. The new column is on disk but unused at
the application layer until v1.2.1.

## What ships in v1.2.1

For reference — not in this release:

- `Fact.source_memory_ids: list[str]` becomes the primary field.
- `agent.facts.trace(fact_id)` returns the full chain of source
  memories and derived facts.
- The SDK starts writing `source_memory_ids` on every new fact; the
  legacy scalar is written through one more cycle for compat.
- A future release (no earlier than v1.3.0) drops the scalar column.

## Running the migration

### Option A — let the agent migrate on first boot

The cleanest path for most users. The migration runs from
`agent.initialize()` exactly the way every prior schema migration has:

```python
agent = WisdomAgent(
    backend=SQLiteBackend("wisdom.db"),
    llm=model,
)
await agent.initialize()  # applies 0024 if not yet applied
```

No diagnostic output beyond the existing "running migrations" log
line. Safe to do on a live store; the migration is non-destructive.

### Option B — run the CLI explicitly

For real fact stores or any deployment where you want pre-flight
stats, dry-run, and explicit confirmation:

```bash
# Pre-flight: report current schema version, count of rows that will
# be backfilled, and the planned operations. No writes.
wisdom-layer-migrate up --dry-run --database sqlite:///wisdom.db

# Apply. Prints per-step progress; stamps a migration row with the
# new schema version.
wisdom-layer-migrate up --database sqlite:///wisdom.db
```

The `--database` argument accepts the same DSNs as `backend_from_url`
(e.g., `sqlite:///path.db`, `postgresql://user:pass@host/db`). The
CLI walks every pending migration in canonical order; if an earlier
migration is missing it will not skip ahead.

### Option C — Postgres deployments

```bash
wisdom-layer-migrate up --database "$WISDOM_LAYER_DATABASE_URL"
```

The Postgres half of `0024_facts_source_ids` uses native `text[]`
rather than a JSON-encoded TEXT column.

## Verifying

After the migration runs, the new column exists on disk and existing
rows are backfilled. A SQL sanity check:

```sql
-- SQLite
SELECT
  COUNT(*)                                            AS total,
  SUM(json_array_length(source_memory_ids) > 0)       AS with_sources,
  SUM(source_memory_id IS NOT NULL)                   AS legacy_populated
FROM facts;
```

`with_sources` should match `legacy_populated` after the migration
runs cleanly. New facts written by v1.2.0 will leave `source_memory_ids`
empty (`[]`) and populate only the legacy scalar — that is the
intended behavior; v1.2.1 turns the dual-write on.

## Application-code changes

**None required for v1.2.0.** Existing reads of `fact.source_memory_id`,
existing writes through `agent.facts.extract()`, and any code that
inspects fact dicts continue to work unchanged.

When you upgrade to v1.2.1, the public API change lands and the
v1.2.1 release notes will cover the small read-side update needed
(prefer `fact.source_memory_ids` over the scalar, iterate when there
are multiple sources).

## Rollback

The migration is reversible within one version:

```bash
wisdom-layer-migrate down --to 0023 --database sqlite:///wisdom.db
```

This recreates the `facts` table without `source_memory_ids` and
restores the v1.1 schema exactly. The legacy `source_memory_id`
column is preserved across the round-trip, so a forward migration
afterwards re-runs the backfill cleanly.

Manual rollback for direct DB editors on SQLite >= 3.35:

```sql
ALTER TABLE facts DROP COLUMN source_memory_ids;
```

(`DROP COLUMN` is unsupported on older SQLite versions; use the CLI
path on those systems.)

## Reporting an issue

If the migration fails on a real store, please email
[jeff@rhatigan.ai](mailto:jeff@rhatigan.ai) with:

- The output of `wisdom-layer-migrate up --dry-run`.
- The schema version reported on failure.
- The first 200 characters of the exception traceback (no PII or
  memory content).

We will not ask for the contents of `facts` or any memory rows.
